# Fall Detection System

## System Architecture

```
[Arduino (Patient Wearable)]
        │  BLE Advertisement (RSSI + state)
        ▼
[scanner.py — Room Scanner(s)]
        │  TCP (patient_id, room, rssi, state)
        ▼
[server.py — Central Server]
        │  WebSocket broadcast
        ▼
[App Clients (dashboard / mobile)]
```

---

## Files

| File | Description |
|---|---|
| `sketch.ino` | Arduino firmware: IMU sampling, TinyML inference, BLE advertising |
| `model_data.h` | TFLite model weights as a compiled C byte array (4208 bytes) |
| `model_settings.h` | Model hyperparameters, quantisation constants, and class labels |
| `scanner.py` | BLE room scanner — detects patient devices and forwards data to the server |
| `server.py` | Central server — receives TCP packets from scanners, tracks patient state, broadcasts WebSocket events |

---

## Hardware Requirements

- **Arduino Nano 33 BLE Sense** (or compatible board with `Arduino_LSM9DS1` IMU and `ArduinoBLE`)
- One or more host machines running `scanner.py` (one per monitored room)
- A server machine running `server.py`

## Arduino Firmware (`sketch.ino`)

### Modes

The firmware operates in three runtime modes, switchable via Serial or BLE write:

| Mode | Command | Description |
|---|---|---|
| `MODE_RECORD` | `r` | Streams raw accelerometer CSV at ~100 Hz for data collection |
| `MODE_CALIBRATE` | `c` | Collects 10 windows to compute per-patient feature normalisation stats |
| `MODE_INFER` | `i` (auto after calibrate) | Runs TinyML inference and advertises results over BLE |

### Inference Pipeline

1. IMU samples accelerometer at **50 Hz**
2. Readings are buffered into a **100-sample sliding window** (90% overlap, advances every 0.2 s)
3. **24 features** are extracted per window (mean, std, min, max, range per axis + magnitude stats)
4. Features are normalised using calibration stats (Welford online algorithm)
5. The TFLite model classifies into one of **7 activity classes**

### Activity Classes

| Index | Label |
|---|---|
| 0 | `walking` |
| 1 | `stumbling` |
| 2 | `idle_standing` |
| 3 | `idle_sitting` |
| 4 | `upstairs` |
| 5 | `downstairs` |
| 6 | `fall` |

### Fall Detection Logic

A fall is confirmed only when **all three** conditions are met:
- Model confidence ≥ **55%** for `fall` class
- Magnitude range (impact sharpness) > **1.40 g**
- Peak acceleration magnitude > **2.35 g**

Post-fall stillness is verified using a **0.20 g** threshold. The fall alert is held active for a configurable countdown of windows.

### BLE Advertisement
The device advertises as `PATIENT_<ID>` and encodes the predicted state index in the manufacturer data payload (`0xFFFF`), allowing passive scanners to read the current activity state without establishing a full BLE connection. A separate `FF01` service UUID is used for passive fall alert detection without requiring an active BLE connection.

**Manufacturer Data Layout (`0xFFFF`)**
| Byte | Description |
|---|---|
| `0` | Predicted activity class index (0–6, matches class labels) |

| Characteristic UUID | Properties | Description |
|---|---|---|
| `19B10001-...` | Read / Notify | Fall alert active flag (0 or 1) |
| `19B10002-...` | Read / Notify | Predicted activity class index |
| `19B10003-...` | Read / Notify | Prediction confidence (0–100) |
| `19B10004-...` | Read / Write | Mode command (ASCII `r` / `c` / `i`) |

### Configuration

To change the device identity, edit the following line in `sketch.ino`:

```cpp
#define DEVICE_NAME "PATIENT_01"
```

To disable TinyML and run without the model:

```cpp
#define ENABLE_TINYML 0
```

## Model Files (`model_data.h` / `model_settings.h`)

Both files are auto-generated by `ml/export_tflite.py` and should not be edited manually.

`model_settings.h` exposes the following constants:

| Constant | Value | Description |
|---|---|---|
| `kFeatureCount` | 24 | Number of input features |
| `kClassCount` | 7 | Number of output classes |
| `kTrainingSampleRateHz` | 50 | IMU sample rate used during training |
| `kWindowSamples` | 100 | Samples per inference window (2 s) |
| `kCalibrationWindowTarget` | 10 | Windows needed for calibration |
| `kInputScale` / `kInputZeroPoint` | — | Input quantisation parameters |
| `kOutputScale` / `kOutputZeroPoint` | — | Output quantisation parameters |
| `kGlobalFeatureMean[24]` | — | Per-feature dataset mean (for normalisation) |
| `kGlobalFeatureStd[24]` | — | Per-feature dataset std (for normalisation) |

`model_data.h` contains the TFLite FlatBuffer as `g_model_tflite[]` (4208 bytes), suitable for use with TensorFlow Lite for Microcontrollers.

## Room Scanner (`scanner.py`)

Each room requires one running instance of `scanner.py`. The scanner uses [Bleak](https://github.com/hbldh/bleak) for async BLE scanning.

### Configuration

Edit the constants at the top of the file:

```python
ROOM_NAME = "ROOM_A"       # Unique identifier for this room
SERVER_IP = "127.0.0.1"    # IP address of the server running server.py
SERVER_PORT = 5000          # Must match server TCP_PORT
PERIODIC_INTERVAL = 5       # Seconds between periodic state updates
STUMBLE_ALERT_SECONDS = 10  # Sustained stumbling threshold for FALL_LIKELY alert
```

### Behaviour

- Passively scans for BLE devices whose name starts with `PATIENT_`
- Reads the activity state index from the manufacturer data payload
- Falls back to UUID-based detection for the `fall` state (`0000ff01-...`)
- **Instant alerts**: sends immediately on `fall` (state index 6) detection
- **Stumble tracking**: if a patient stumbles for more than `STUMBLE_ALERT_SECONDS`, sends a `FALL_LIKELY` alert
- **Periodic updates**: sends the latest known state for all detected patients every `PERIODIC_INTERVAL` seconds

### Installation

```bash
pip install bleak
python scanner.py
```

## Central Server (`server.py`)

### Ports

| Port | Protocol | Purpose |
|---|---|---|
| `5000` | TCP | Receives packets from room scanners |
| `5001` | WebSocket | Broadcasts events to app clients |

### Incoming TCP Packet Format (from scanner)

```
patient_id,room_name,rssi,state_index
PATIENT_01,ROOM_A,-72,0

# Special case for fall likely:
patient_id,room_name,rssi,FALL_LIKELY
```

### Outgoing WebSocket Packet Formats

| Type | Trigger | Fields |
|---|---|---|
| `snapshot` | New client connects | Full state of all known patients |
| `state_change` | Patient state changed | `patient_id`, `room`, `rssi`, `state_index`, `state`, `location`, `name` |
| `heartbeat` | Periodic update, state unchanged | Same as `state_change` |
| `fall` | State index 6, debounced | `patient_id`, `room`, `name`, `contacts` |
| `fall_likely` | `FALL_LIKELY` received from scanner | `patient_id`, `room`, `name` |

### Incoming WebSocket Message (from app)

Apps can register patient metadata:

```json
{
  "type": "register",
  "patient_id": "PATIENT_01",
  "name": "Jane Doe",
  "room": "ROOM_A",
  "contacts": ["07700 900000"]
}
```

### Key Parameters

| Constant | Value | Description |
|---|---|---|
| `FALL_LATCH_SECONDS` | 60 s | Suppresses duplicate fall alerts within this window |
| `STALE_THRESHOLD` | 30 s | RSSI readings older than this are ignored for room localisation |

### Installation

```bash
pip install websockets
python server.py
```

## Quick Start

1. **Flash the Arduino** with `sketch.ino`, `model_data.h`, and `model_settings.h`. Set `DEVICE_NAME` appropriately.
2. **Start the server** on the central machine:
   ```bash
   python server.py
   ```
3. **Start a scanner** in each room, updating `ROOM_NAME` and `SERVER_IP` for each:
   ```bash
   python scanner.py
   ```
4. **Power on the Arduino**. Open Serial Monitor at 115200 baud.
5. Send `c` over Serial (or BLE) to begin **calibration**. Walk normally until complete.
6. The device automatically switches to **inference mode** and begins broadcasting state over BLE.
7. Connect an app client to `ws://<server_ip>:5001` to receive live events.

---

## Notes

- The Arduino LED is **on** when a BLE central is connected and **off** otherwise. Rapid flashing indicates an IMU or model initialisation failure.
- In `MODE_RECORD`, the Serial output is CSV (`timestamp_ms,ax,ay,az`) and can be piped directly into a data collection script.
- Each `scanner.py` instance must have a unique `ROOM_NAME`; patient location is determined server-side by the strongest RSSI across all rooms.
